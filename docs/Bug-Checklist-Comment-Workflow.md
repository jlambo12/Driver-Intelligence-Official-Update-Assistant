# Инструкция: как заново запустить баг-чеклист и сразу оставлять комментарии

Дата: 2026-04-05  
Аудитория: QA/Dev/Validation внутри проекта DriverGuardian.

## Когда использовать

Используй этот процесс, когда нужно:
- быстро поднять новый цикл проверки на баги;
- фиксировать баги единообразно;
- сразу оставлять комментарии по ходу проверки (чтобы не потерять контекст);
- связать результат проверки с evidence-доками и owner-issue.

---

## 1) Подготовь «папку прохода» и контекст

1. Определи дату прохода (например, `2026-04-05`).
2. Проверь актуальные документы:
   - `docs/Windows-Validation-Matrix.md` — сценарии и критерии PASS/DEGRADED/FAIL;
   - `docs/validation-evidence/windows/<дата>/README.md` — куда складывать результаты;
   - `docs/Dev-Pilot-Bug-Log-Template.md` — шаблон карточки бага.
3. Создай (или обнови) файл заметок прохода, например:
   - `docs/validation-evidence/windows/<дата>/CHECKLIST-RUN.md`.

Рекомендуемый заголовок в файле прохода:
- кто проверяет;
- build/commit;
- среда (Windows версия, Online/Offline, права пользователя);
- список сценариев на сегодня.

---

## 2) Скопируй и заполни базовый чеклист

Для каждого сценария в `CHECKLIST-RUN.md` используй короткий блок:

```md
## Scenario EV-XXX: <название>
- Статус: TODO / IN-PROGRESS / PASS / DEGRADED / FAIL
- Environment:
- Preconditions:
- Steps:
  1.
  2.
  3.
- Ожидание:
- Факт:
- Артефакты:
  - лог:
  - скрин:
  - report:
- Комментарий в процессе:
```

Практика: не ждать конца сценария — сразу обновляй строку «Комментарий в процессе» после каждого существенного шага.

---

## 3) Если найден баг — сразу заводи карточку по шаблону

1. Возьми шаблон из `docs/Dev-Pilot-Bug-Log-Template.md`.
2. Создай отдельный файл бага:
   - `docs/validation-evidence/windows/issues/WIN-VAL-YYYY-MM-DD-XX.md`.
3. Минимум, который должен быть заполнен сразу:
   - Bug ID;
   - Severity/Priority;
   - Expected vs Actual;
   - шаги воспроизведения;
   - correlation_id / event_code;
   - owner.

Важно: если статус сценария DEGRADED/FAIL, issue-файл обязателен в тот же день.

---

## 4) Как «сразу делать комментарии», чтобы не терять ход мыслей

Используй правило «комментарий в 3 слоя»:

1. **Операционный комментарий (в чеклисте):**
   - что сделал только что;
   - что получил;
   - что делаю следующим шагом.

2. **Технический комментарий (в issue-файле):**
   - наблюдение (факт);
   - гипотеза причины;
   - какие логи/артефакты это подтверждают.

3. **Триаж-комментарий (в issue/PR обсуждении):**
   - impact для пользователя;
   - риск безопасности/доверия к source;
   - решение: fix now / defer / acceptable degraded.

Шаблон короткого комментария:

```text
[HH:MM UTC] Scenario EV-XXX, шаг N:
Факт: ...
Ожидание: ...
Разница: ...
Следующее действие: ...
Артефакт: <путь к логу/скрину>
```

---

## 5) Ежедневный ритм (рекомендуется)

- Старт дня: пометить `IN-PROGRESS` сценарии.
- Во время прогона: комментарии в реальном времени (не в конце).
- Конец дня:
  - обновить статусы PASS/DEGRADED/FAIL;
  - убедиться, что для всех DEGRADED/FAIL есть issue + owner;
  - синхронизировать краткий итог в `README.md` текущей даты.

---

## 6) Definition of Done для прохода

Проход считается оформленным корректно, если:
1. У каждого сценария есть финальный статус;
2. У каждого статуса DEGRADED/FAIL есть issue-файл в `docs/validation-evidence/windows/issues/`;
3. В issue есть owner и план валидации после фикса;
4. В чеклисте есть минимум один комментарий «по ходу» на сценарий;
5. Есть ссылки на артефакты (лог/скрин/репорт).

---

## 7) Быстрый старт (копипаст)

```text
1) Создать docs/validation-evidence/windows/<дата>/CHECKLIST-RUN.md
2) Скопировать сценарии из Windows-Validation-Matrix
3) Для каждого сценария вести статус + комментарий в реальном времени
4) При первом подтверждении бага создать WIN-VAL-... issue-файл
5) В конце дня сверить: все DEGRADED/FAIL имеют owner и issue-link
```

Эта инструкция специально сделана как «повторяемый процесс», чтобы можно было быстро восстановить поток, даже если прошлую ветку/обсуждение не удалось найти.
